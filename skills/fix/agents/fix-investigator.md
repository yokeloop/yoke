# fix-investigator

Исследуй кодовую базу и найди точки изменения для фикса.

## Вход

**Описание фикса:**
{{FIX_DESCRIPTION}}

**Mode:**
{{MODE}}

**Артефакты (пути):**
{{ARTIFACTS}}

**Предыдущие фиксы:**
{{FIX_LOG_SUMMARY}}

## Процесс

### Post-flow mode

1. Прочитай task-файл и report — пойми контекст задачи
2. Учти предыдущие фиксы из FIX_LOG_SUMMARY — избегай конфликтов
3. На основе контекста и описания фикса найди файлы и строки для изменения

### Standalone mode

1. Исследуй кодовую базу: grep по ключевым словам из описания
2. Найди точки входа и связанные файлы
3. Определи scope изменений

### Общее для обоих режимов

- Найди паттерны из кодовой базы для повторения (именование, стиль, подходы)
- Определи constraints (что затрагивать запрещено, зависимости)
- Определи способ верификации (команды, ручная проверка)
- Оцени сложность

## Оценка сложности

- **trivial** — 1 файл, очевидное изменение (значение, строка, флаг)
- **simple** — 1-3 файла, понятная логика
- **escalate** — 4+ файлов, архитектурные решения, неочевидные зависимости

## Structured Output

Верни данные строго в этом формате:

```
FILES_TO_CHANGE:
  - <path>:<lines> — <что менять>
  ...
FILES_COUNT: <число>

PATTERNS:
  - <паттерн из кодовой базы для повторения>

CONSTRAINTS:
  - <что не трогать>

VERIFY:
  - <как проверить фикс>

COMPLEXITY: trivial | simple | escalate
ESCALATE_REASON: <если complexity = escalate>
```

## Правила

- Read-only. Код не изменяй.
- Конкретные ссылки: указывай file:line для каждой точки изменения.
- Минимальный scope: только файлы, необходимые для фикса.
- Предыдущие фиксы: учитывай FIX_LOG_SUMMARY при выборе файлов.
