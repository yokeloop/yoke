# Подготовка отчёта для code review

Ты помогаешь с подготовкой отчёта для code review.
Анализируешь все изменения относительно origin/main и пишешь структурированный отчёт.

---

## Вход

`$ARGUMENTS` — task-slug или путь к task-файлу.

Если не указан — определи slug из текущей ветки или последнего `docs/ai/*/` каталога.

---

## Процесс

### Шаг 1 — Собери данные

```bash
# Коммиты
git log origin/main..HEAD --oneline

# Изменённые файлы со статистикой
git diff origin/main...HEAD --stat

# Полный diff
git diff origin/main...HEAD
```

Извлеки `SLUG` из коммитов (scope в conventional commit) или из `docs/ai/<slug>/`.

### Шаг 2 — Проанализируй изменения

На основе diff:

1. **Контекст и цель** — что сделано и зачем (1-3 предложения). Прочитай task-файл `docs/ai/<SLUG>/<SLUG>-task.md` если существует.

2. **Ключевые участки кода для ревью** — максимум 10 пунктов. Для каждого: файл (функция/класс), почему важен.

3. **Сложные или нетривиальные решения** — где логика сложная, есть trade-off'ы или спорные решения. Для каждого: почему так сделано, что ревьюеру проверить. Приведи примеры кода.

4. **Вопросы к ревьюеру** — конкретные вопросы по архитектуре, контрактам, перформансу, безопасности, читаемости.

5. **Риски и влияние** — перформанс, миграции, совместимость, безопасность. На что обратить особое внимание.

6. **Тесты и ручные проверки** — какие авто-тесты нужны, какие ручные сценарии выполнить.

7. **Out of scope** — что намеренно оставлено вне изменений, чтобы не раздувать PR.

### Шаг 3 — Запиши отчёт

`docs/ai/<SLUG>/<SLUG>-review.md`:

````markdown
# Code Review: <SLUG>

### Контекст и цель

<1-3 предложения>

### Коммиты

| Hash    | Описание        |
| ------- | --------------- |
| abc1234 | feat(slug): ... |
| ...     | ...             |

### Изменённые файлы

| Файл             | +/-     | Описание         |
| ---------------- | ------- | ---------------- |
| src/path/file.ts | +42/-10 | <что изменилось> |
| ...              | ...     | ...              |

### Ключевые участки для ревью

1. **`src/path/file.ts:fn()`** — <почему важен>
2. ...

### Сложные решения

1. **<Что>** (`src/path/file.ts:42`)
   <Почему так сделано. Trade-off'ы.>
   ```ts
   // пример кода
   ```
````

2. ...

### Вопросы к ревьюеру

1. <Конкретный вопрос>
2. ...

### Риски и влияние

- <Риск>: <на что обратить внимание>
- ...

### Тесты и ручные проверки

**Авто-тесты:**

- <что должно быть покрыто>

**Ручные сценарии:**

1. <Шаг> → <ожидаемый результат>
2. ...

### Out of scope

- <Что оставлено за рамками и почему>

```

### Шаг 4 — Сообщи пользователю

```

Review записан: docs/ai/<SLUG>/<SLUG>-review.md

```

---

## Правила

- Формат: секции с заголовками ###, краткие буллет-списки, без лишних объяснений
- Не переписывать код — только ссылки на файлы и строки
- Примеры кода — только в секции "Сложные решения" для пояснения логики
- Язык отчёта — язык коммитов и task-файла
- Пиши активным залогом, конкретным языком, без лишних слов
```
