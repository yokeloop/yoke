# Дизайн: Переписать скилл /task — prompt-инженер вместо конвертера тикетов

**Дата:** 2026-03-12
**Статус:** утверждён

## Суть

Переписать `skills/task/SKILL.md` — из простого конвертера тикетов в prompt-инженера, который исследует проект и формирует максимально эффективный промт для AI-реализации задачи.

## Роль

Технический prompt-инженер. Исследует проект, понимает что и как реализовать, формулирует задачу так, чтобы реализатор получил полный контекст с первой попытки. НЕ реализует сам.

## Вход

`$ARGUMENTS` содержит одно или оба:
1. **URL тикета** — GitHub Issues, YouTrack, Jira и др.
2. **Текст задачи** — описание, примеры кода, ссылки на материалы

## Выход

Файл `docs/ai/<task-slug>/<task-slug>-task.md`

## Процесс — 4 фазы

### Фаза 1: Parse

1. **Получить содержимое тикета:**
   - GitHub Issues → `gh issue view <url>`
   - Остальные трекеры → URL как якорь для task-slug, содержимое из текста пользователя

2. **Извлечь и сохранить материалы** (до любого анализа):
   - Ссылки (Figma, документация, API-эндпоинты)
   - Пути к скриншотам/изображениям
   - Вставленные куски кода
   - Внешние ресурсы (gist, confluence, wiki)
   - Список неприкосновенен — попадёт в секцию «Материалы» как есть

3. **Сформировать task-slug:**
   - С URL: `<ID>-<описание>` (86-black-jack-page, R2-50-user-id-to-db)
   - Без URL: `<описание>` (fix-navbar-overflow)
   - Всегда english kebab-case, 2-4 слова в описании

### Фаза 2: Investigate

Всегда использовать все доступные инструменты по максимуму:

1. **Grep/Glob** — найти все связанные файлы, тесты, конфиги
2. **Read** — прочитать затронутые файлы, понять текущую реализацию
3. **Agent (code-explorer)** — проследить execution path, зависимости, паттерны
4. **Agent (code-architect)** — архитектура затронутой области, data flow, точки интеграции

Что искать:
- Какие файлы/функции будут затронуты (пути, номера строк)
- Паттерны и конвенции в затронутой части кодовой базы
- Тесты, покрывающие эту область
- Что может сломаться

### Фаза 3: Synthesize

Применить 5 измерений к находкам:

1. **Intent Clarity** — однозначность намерения. Конкретные действия вместо размытых глаголов.
2. **Scope Boundaries** — что входит в задачу, что НЕ входит. Файлы/модули в скоупе.
3. **Context Anchoring** — конкретные пути, функции, строки из фазы Investigate.
4. **Acceptance Criteria** — команды проверки, ожидаемое поведение, edge cases.
5. **Output Format** — что ожидается от реализатора (план+код, только код, тесты).

Сгенерировать уточняющие вопросы (3-10) на основе пробелов:
- Чеклист с 2-3 вариантами + плейсхолдер для своего
- Вопросы про реализацию, не про постановку

Классифицировать сложность (trivial/simple/medium/complex) — для информации реализатору.

### Фаза 4: Write

Записать файл `docs/ai/<task-slug>/<task-slug>-task.md` по шаблону:

```markdown
# <Заголовок задачи>

**Тикет:** <URL или "—">
**Сложность:** <trivial | simple | medium | complex>

## Task

<Одно чёткое предложение — что сделать>

## Context

<Архитектура, ключевые файлы с путями и строками, паттерны>

## Requirements

<Пронумерованные, проверяемые требования>

## Constraints

<Что НЕ менять, что не ломать, какие подходы избегать>

## Verification

<Команды, ожидаемое поведение, edge cases>

## Уточняющие вопросы

1. **Вопрос?**
   - [ ] Вариант A
   - [ ] Вариант B
   - [ ] Свой вариант: ___

## Материалы

- [Описание](url) — что это
- `path/to/file` — что это
```

## Решения

- **Подход:** Structured pipeline (Parse → Investigate → Synthesize → Write)
- **Материалы:** Извлекаются в фазе Parse до анализа — не теряются
- **Инструменты:** Всегда максимум, без градации по сложности
- **Сложность:** Указывается в выходном файле для реализатора
- **Без URL:** task-slug только из описания (english kebab-case)
- **Декомпозиция:** Нет — одна задача = один файл (декомпозиция в будущем /brain)
- **5 измерений:** Ссылаемся на концепции, не копируем определения целиком
